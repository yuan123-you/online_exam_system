## 二、审计发现详情

### SEC-001 | 认证拦截器缺乏用户身份验证（严重）

| 字段 | 内容 |
|------|------|
| **漏洞编号** | SEC-001 |
| **严重级别** | Critical |
| **漏洞类型** | 认证绕过 / 身份伪造 |
| **影响文件** | `backend/src/main/java/com/onlineexam/config/AuthInterceptor.java` |
| **CVSS 评分** | 9.8 |

**漏洞描述**:

`AuthInterceptor` 仅检查请求头 `X-User-Id` 是否存在且非空，**不验证该用户 ID 是否对应数据库中的真实用户**。攻击者可以构造任意 `X-User-Id` 请求头（例如管理员的用户 ID），从而冒充任何用户身份访问系统。

由于系统的认证完全依赖 `X-User-Id` 请求头（无 JWT 签名、无 Session），且用户 ID 格式为可预测的 `prefix-timestamp-hex` 模式，一旦攻击者获取任何有效用户 ID，即可完全冒充该用户。

**攻击场景**:
1. 攻击者通过 XSS 或日志泄露获取管理员用户 ID（如 `admin-1718234567890-a1b2c3`）
2. 构造请求：`curl -H "X-User-Id: admin-1718234567890-a1b2c3" http://target/api/entities`
3. 成功以管理员身份执行操作

**修复方案（已实施）**:

在 `AuthInterceptor` 中增加用户存在性验证，使用 30 秒短期缓存避免每次请求都读取数据库：

```java
// 验证用户 ID 是否对应数据库中的真实用户
private boolean isUserValid(String userId) {
    long now = System.currentTimeMillis();
    Long expiry = validUserCache.get(userId);
    if (expiry != null && expiry > now) return true;
    boolean exists = storeService.readStore().users.stream()
        .anyMatch(u -> Objects.equals(String.valueOf(u.get("id")), userId));
    if (exists) validUserCache.put(userId, now + CACHE_TTL_MS);
    return exists;
}
```

**修复后效果**: 伪造的用户 ID 将被拒绝（返回 401），只有数据库中存在的用户 ID 才能通过拦截器。

### SEC-002 | AI API 密钥硬编码在配置文件中（严重）

| 字段 | 内容 |
|------|------|
| **漏洞编号** | SEC-002 |
| **严重级别** | Critical |
| **漏洞类型** | 敏感数据暴露 / 硬编码凭据 |
| **影响文件** | `backend/src/main/resources/application.yml` |
| **CVSS 评分** | 9.1 |

**漏洞描述**:

智谱 AI GLM-4-Flash API 密钥以明文形式硬编码为 `application.yml` 中的默认回退值：

```yaml
ai:
  api-key: ${AI_API_KEY:82f20bac24a545c595dc30f7eee3dfd1.eJmxuTgiNAxaDNEX}
```

该文件会被提交到版本控制系统，导致 API 密钥泄露。任何有代码仓库访问权限的人都可以获取该密钥，可能导致：
- 未经授权的 AI API 调用，产生费用
- API 密钥被滥用或转售
- 如果密钥关联了账号信息，可能暴露更多敏感数据

**修复方案（已实施）**:

移除默认回退值，强制通过环境变量配置：

```yaml
ai:
  api-key: ${AI_API_KEY:}
```

**部署须知**: 生产环境必须设置环境变量 `AI_API_KEY`，否则 AI 功能将不可用。

### SEC-003 | AI 解析接口缺乏角色验证（高危）

| 字段 | 内容 |
|------|------|
| **漏洞编号** | SEC-003 |
| **严重级别** | High |
| **漏洞类型** | 授权缺失 |
| **影响文件** | `backend/src/main/java/com/onlineexam/service/AiService.java` |
| **CVSS 评分** | 7.5 |

**漏洞描述**:

`AiService.explainAnswer()` 方法没有进行任何角色验证。虽然 `AuthInterceptor` 会拦截未认证请求，但该方法未检查用户是否为数据库中的有效用户实体（与其他 AI 端点不同，其他端点都有 `isRole()` 检查）。

如果传入的 `userId` 在数据库中不存在但通过了拦截器（在 SEC-001 修复前），则可能触发异常或导致未授权的 AI API 调用。

**修复方案（已实施）**:

添加用户存在性验证：

```java
Map<String, Object> user = find(store.users, userId);
if (user == null) return error(HttpStatus.UNAUTHORIZED, "Not authenticated.");
```

### SEC-004 | 题目恢复接口所有权校验绕过（高危）

| 字段 | 内容 |
|------|------|
| **漏洞编号** | SEC-004 |
| **严重级别** | High |
| **漏洞类型** | 水平越权 (IDOR) |
| **影响文件** | `backend/src/main/java/com/onlineexam/service/QuestionService.java` |
| **CVSS 评分** | 7.2 |

**漏洞描述**:

`QuestionService.restoreQuestions()` 方法在恢复软删除题目时存在所有权校验绕过：

1. `store.questions` 仅包含 `deleted=0` 的题目（非删除状态）
2. 对于已软删除的题目（`deleted=1`），在 `store.questions` 中找不到
3. 原代码在题目未找到时直接跳过所有权检查，调用 `restoreRecord()`
4. 导致任何教师都可以恢复其他教师的软删除题目

**攻击场景**:
- 教师 A 删除了一道题目（ID: `question-123`）
- 教师 B 调用 `/api/questions/batch-restore`，传入 `{"ids": ["question-123"]}`
- 教师 B 成功恢复了教师 A 的题目，实现了水平越权

**修复方案（已实施）**:

1. 在 `QuestionRepository` 中新增 `findTeacherIdIncludingDeleted()` 方法，可查询包括软删除题目的教师 ID
2. 在 `restoreQuestions()` 中，对软删除题目使用数据库级别的所有权校验

```java
String teacherId = questionRepository.findTeacherIdIncludingDeleted(id);
if (teacherId == null) {
    errors.add(mapOf("id", id, "message", "Question not found."));
    continue;
}
if (!Objects.equals(teacherId, userId)) {
    errors.add(mapOf("id", id, "message", "Question not owned by you."));
    continue;
}
```

### SEC-005 | 数据库凭据硬编码在配置文件中（高危）

| 字段 | 内容 |
|------|------|
| **漏洞编号** | SEC-005 |
| **严重级别** | High |
| **漏洞类型** | 敏感数据暴露 / 硬编码凭据 |
| **影响文件** | `backend/src/main/resources/application.yml` |
| **CVSS 评分** | 7.0 |

**漏洞描述**:

MySQL 数据库默认密码 `123456` 硬编码在 `application.yml` 中：

```yaml
spring:
  datasource:
    password: ${MYSQL_PASSWORD:123456}
```

虽然使用了环境变量语法，但默认回退值 `123456` 会被提交到代码仓库，且是一个极弱的密码。

**修复方案（已实施）**:

移除默认回退值：

```yaml
spring:
  datasource:
    password: ${MYSQL_PASSWORD:}
```

**部署须知**: 生产环境和开发环境必须设置环境变量 `MYSQL_PASSWORD`。

### SEC-006 | 登录接口缺乏暴力破解防护（中等）

| 字段 | 内容 |
|------|------|
| **漏洞编号** | SEC-006 |
| **严重级别** | Medium |
| **漏洞类型** | 暴力破解 / 缺乏速率限制 |
| **影响文件** | `backend/src/main/java/com/onlineexam/service/AuthService.java` |
| **CVSS 评分** | 5.3 |

**漏洞描述**:

`AuthService.login()` 方法没有登录失败次数限制。攻击者可以对登录接口进行暴力破解攻击，尝试大量用户名/密码组合。虽然 AI 端点有速率限制，但登录端点没有。

**修复方案（已实施）**:

添加基于用户名的登录失败计数器和临时锁定机制：
- 5 次失败尝试后锁定账户 5 分钟
- 成功登录后清除失败计数
- 使用 `ConcurrentHashMap` 保证线程安全

```java
private final ConcurrentHashMap<String, List<Long>> loginAttempts = new ConcurrentHashMap<>();
private static final int MAX_FAILED_ATTEMPTS = 5;
private static final long LOCKOUT_WINDOW_MS = 5 * 60 * 1000; // 5 minutes
```

### SEC-007 | 教师更新实体时 teacherId 可被篡改（低危）

| 字段 | 内容 |
|------|------|
| **漏洞编号** | SEC-007 |
| **严重级别** | Low |
| **漏洞类型** | 数据完整性 |
| **影响文件** | `backend/src/main/java/com/onlineexam/service/EntityCrudService.java` |
| **CVSS 评分** | 3.1 |

**漏洞描述**:

`EntityCrudService.updateEntity()` 方法在更新实体时，使用 `next.putAll(record)` 将请求体的所有字段合并到现有记录中。教师可以在请求体中传入不同的 `teacherId`，将其题目的所有权转移给其他教师。

虽然此操作需要教师已认证且只能操作自己的题目，但转移所有权后：
- 原教师失去对题目的控制权
- 目标教师获得不想要的题目
- 审计日志的归属关系混乱

**修复方案（已实施）**:

在更新操作后强制重置教师的资源 `teacherId` 为当前用户：

```java
if (isRole(user, "teacher") && Set.of("questions", "papers", "exams").contains(entity)) {
    next.put("teacherId", userId);
}
```

## 三、已验证安全区域（无需修复）

### 3.1 SQL 注入防护

**状态**: 安全

所有数据库操作均使用 `JdbcTemplate` 的参数化查询（`?` 占位符），包括：
- `StoreService.java` 中所有 `jdbc.update()` 和 `jdbc.queryForList()` 调用
- `QuestionRepository.java`、`ExamRepository.java` 等仓库类
- `queryQuestionsPage()` 中的动态 SQL 拼接使用了参数化 `where` 子句

未发现任何字符串拼接构建 SQL 的情况。

### 3.2 XSS 防护

**状态**: 安全

- Vue 模板默认转义所有 `{{ }}` 插值
- `v-html` 仅在 `AppShell.vue` 中使用，且数据源为硬编码的 SVG 图标映射表（`iconMap`），不涉及用户输入
- 已配置 `Content-Security-Policy` 响应头
- 已配置 `X-XSS-Protection: 1; mode=block` 响应头

### 3.3 CSRF 防护

**状态**: 可接受

- 系统使用 `X-User-Id` 自定义请求头进行认证，非 Cookie 认证
- CORS 配置限制了允许的来源域名（localhost、ccwu.cc 域名、特定 IP）
- 未使用 `allowedOrigins("*")`

### 3.4 密码安全

**状态**: 安全

- 使用 `BCryptPasswordEncoder` 加密存储密码
- 登录响应中不包含密码字段（`sanitizeUser()` 已移除）
- Bootstrap 响应中所有用户数据均经过 `sanitizeUser()` 处理
- 密码修改需要验证旧密码
- 密码长度限制为 6-100 字符

### 3.5 数据隔离

**状态**: 安全（修复后）

- `BootstrapService` 对不同角色返回不同范围的数据
- 学生只能看到自己的数据
- 教师只能看到自己创建的数据
- 错题本控制器验证 `studentId` 所有权
- 考试控制器验证 `teacherId` 所有权
- `DataIsolationService` 提供了完整的数据隔离检查方法

### 3.6 AI API 安全

**状态**: 安全（修复后）

- API 密钥从环境变量读取（修复后）
- 所有 AI 端点有速率限制（每分钟 10 次）
- 所有 AI 端点需要认证
- AI 响应经过 JSON 解析后才返回给前端（非直接透传原始文本）
- RestTemplate 配置了 15 秒超时

### 3.7 前端安全

**状态**: 安全

- 前端无硬编码的 API 密钥或敏感凭据
- 认证令牌仅存储在内存变量中（非 localStorage/cookies）
- 路由守卫正确实现了角色访问控制
- 无敏感逻辑在前端执行

## 五、未来安全改进建议

### 5.1 认证架构升级（优先级：Critical）

当前 `X-User-Id` 请求头认证方案存在根本性弱点。建议升级为：

1. **JWT Token 认证**: 登录成功后签发 JWT Token，包含用户 ID、角色、过期时间，使用 HMAC-SHA256 签名。前端存储 Token 并在请求头中发送 `Authorization: Bearer <token>`。
2. **Session 认证**: 使用 Spring Session 或 Redis 存储会话，前端通过 Cookie 或请求头传递 Session ID。
3. **Token 刷新机制**: 短期 Access Token + 长期 Refresh Token 组合。

### 5.2 输入验证增强（优先级：Medium）

- 添加全局请求参数验证（如 `@Valid` 注解 + `Validation`）
- 对批量操作添加最大数量限制（如批量导入最多 1000 条记录）
- 对分页参数添加上限（如 `pageSize` 最大 100）

### 5.3 审计日志增强（优先级：Medium）

- 记录所有认证失败事件（包括 IP 地址）
- 记录所有权限拒绝事件
- 记录敏感操作（密码修改、用户删除、批量数据操作）
- 实现日志告警机制

### 5.4 传输安全（优先级：High）

- 强制 HTTPS（在 Nginx/反向代理层配置）
- 配置 HSTS（HTTP Strict Transport Security）响应头
- 禁用 HTTP/1.0

### 5.5 依赖安全扫描（优先级：Medium）

- 集成 OWASP Dependency Check 或 Snyk 到 CI/CD 流程
- 定期更新依赖版本
- 监控已知漏洞数据库

### 5.6 考试安全增强（优先级：Medium）

- 添加 IP 绑定（限制学生只能从特定 IP 参加考试）
- 实现考试期间的 WebSocket 实时监控
- 添加更复杂的反作弊检测（如切屏检测增强、复制粘贴禁用）

### 5.7 性能与安全平衡（优先级：Low）

- 为 `AuthInterceptor` 的用户验证缓存添加最大条目限制，防止内存泄漏
- 为登录失败计数器添加自动清理机制
- 考虑使用 Redis 替代内存缓存以支持多实例部署

## 六、结论

本次安全审计发现并修复了 7 个安全漏洞，其中 2 个严重级别、3 个高危级别。所有修复已通过编译验证。

**最关键的安全风险** 是认证拦截器缺乏用户身份验证（SEC-001），该漏洞允许攻击者通过伪造请求头冒充任意用户。此漏洞已通过在拦截器中添加用户存在性验证修复。

**长期建议** 是将认证架构升级为 JWT Token 方案，从根本上解决 `X-User-Id` 请求头可被伪造的问题。
